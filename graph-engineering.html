Graph Engineering

当 Agent 不再是一个"自己转圈的循环"，而是一张"你先画好的图"。

FieldAgent 系统架构
FoilLoop Engineering
State2026 上半年由框架侧推动为共识
Read time~8 分钟

过去两年，做 AI Agent 的默认写法是——把模型丢进一个 while 循环里，让它自己看状态、自己选工具、自己判断"是不是完成了"。这套东西被叫做 Loop Engineering，也是 ChatGPT tool-use、ReAct、AutoGPT、Claude Code、Cursor Agent 的共同骨架。

但从 2026 年初开始，一个更工程化的说法在框架侧站稳脚跟：Graph Engineering——把 Agent 写成一张显式的计算图，节点是模型调用/工具/检索，边是控制流和数据流，工程师负责画图，模型负责填空。这不是给循环换皮，而是控制权的一次搬家。

§01什么是 Graph Engineering

一句话：把 Agent 的控制流从模型手里拿出来，交回给工程师，用一张有向图显式表达。

图上的每个节点是一个明确职责的原子——"起草回复"、"调检索"、"跑一段 Python"、"叫审核"；边是"这个节点跑完之后去哪个"，可以是硬编码的顺序、条件分支、并行 fan-out、循环回退，也可以是模型基于状态输出的路由决定。整张图有一个共享的、可 checkpoint 的 State 对象，节点读写它、图引擎推进它。

这套形状最早在 LangGraph、Pocketflow、Burr、Google ADK、Semantic Kernel Process 这些框架里被抽象出来，但真正让它变成一个"名字"、能被 CTO 挂在嘴边说的，是 2026 年上半年一系列生产事故的反思——大家发现纯循环 Agent 在长任务里，失败模式几乎不可复现，也几乎不可回归测试。

Graph Engineering 的核心动作有四条：

①拆节点。把原本塞进一个大 prompt 的活儿，拆成职责单一、可单测的节点；每个节点自己就是一个可以离线跑的函数。

②画边。确定的顺序就写硬边，模棱两可的分支才交给"路由节点"让模型二选一——不是让模型决定"下一步做什么"，而是"从这几条已列好的路里挑一条"。

③管状态。一个显式的 State schema，谁读谁写写得明明白白；每一步都能 checkpoint，出错能从任意节点续跑。

④插人类。Human-in-the-loop 成为一等公民——图可以在任意节点暂停等审批，而不是"把循环打断"这种脏活。

§02Loop Engineering 是怎么回事

Loop Engineering 不是一个坏词，它就是我们过去两年默认在做的事——ReAct 论文之后一路演化下来的主流范式：给模型一个 system prompt、一堆工具描述、一段用户任务，然后进入一个 while 循环，每一轮模型输出"我要调哪个工具、参数是什么"，宿主执行、把结果塞回上下文，再问一次，直到模型说"done"。

它的力量在于极致的灵活——面对任何没见过的任务形状，它都能"想办法"；不需要提前建图，不需要写编排代码。Claude Code、Cursor Agent、Cline、Devin 都是这个骨架的极致优化版，靠的是超大上下文、系统提示词的手艺、以及模型自己的规划能力。

但它也带来了那种熟悉的痛：

长任务会漂——模型走着走着就忘了自己在干什么，或者把已经完成的事情又做一遍。失败不可复现——同一个输入，两次跑出两条不同的路径，日志翻半天找不到"为什么在第 7 步选了这个工具"。回归测试几乎无法建立——你没法给一条"未来会出现"的执行轨迹写单元测试。产线的稳定性靠 prompt 手艺撑着，人一走 prompt 就没人敢改。

这些痛不是循环范式的 bug，是它的 feature 的另一面——你选择了"让模型自由"，就同时选择了"让工程性让位"。

"Loop 是把方向盘交给模型；Graph 是把地图交给模型，方向盘还在工程师手里。" — 编者按

§03十项对比：Loop vs Graph

下面这张表只对比两者的工程属性，不评价谁更"聪明"——同一个模型，装在两种骨架里，工程体验截然不同。

Table 01 · Loop Engineering vs Graph Engineering — 十项工程侧对比
维度 ALoop Engineering BGraph Engineering
控制流模型每步自己决定下一步做什么。工程师画好节点与边，模型只在节点内做局部决策，或从预设路径里挑一条。
状态管理状态藏在上下文窗口里，随对话滚动。显式的 State schema，可读可写、可 checkpoint、可回滚。
可预测性低——同输入两次可能走两条路。高——图形状固定，路径可枚举、可回放。
可调试性看整段 trace，靠人眼找漂移点。逐节点看输入输出，State diff 一目了然。
并行能力顺序执行为主，靠模型自己想到"能并行"很难。fan-out 是图原语，独立子图可以天然并发。
版本化版本 = prompt 文本 + 工具集。版本 = 图拓扑 + 每个节点的 prompt + State schema。
Human-in-the-loop需要"打断循环"这种脏 hack。一等公民——任何节点都能暂停等审批。
回归测试难——路径无法枚举。可以给每个节点单独写测试，也能对完整图做端到端 replay。
失败模式漂移、死循环、无限重试、忘记目标。节点级失败，边界清晰，重试语义明确。
擅长场景探索性任务、编码副驾、开放问答、任务形状事先未知。已知形状的生产工作流、审批链、多方协作、合规敏感场景。

§04为什么是
现在火起来

产线在还债

2024–2025 一轮 Agent 上线潮后，2026 年是"回头修 bug"的一年——SRE 拿着日志问"为什么这个 case 模型走了这条路"，Loop 给不出答案，Graph 至少能给出 State diff。

模型不再是稀缺资源

推理成本从"每次调用心疼一下"降到"随便打"，一个节点里塞两次 LLM 校验也不心疼；这才让"细粒度节点"成为可能，之前 Loop 是被成本逼出来的粗粒度。

合规接入

金融/医疗/政府场景要求"每一步为什么这么做"可追溯；Loop 的"模型自己决定"根本进不去这些门，Graph 的显式路径是准入的必要条件。

框架就位

LangGraph、Google ADK、Semantic Kernel Process、Pocketflow、Burr 在 2025 年下半年到 2026 年初集中成熟，让画图的工程成本从"自己造轮子"降到"import 一下"。

§05什么时候用哪个

选 Loop——当你面对的任务形状每次都不一样，或者根本还不知道会遇到什么。写代码副驾、开放问答、研究型探索、一次性脚本、还在做需求 discovery 的 MVP。这时候你要的是模型的自主性，不是可预测性。硬去建图只会把不确定性提前锁死在错的地方。

Claude Code / Cursor Agent 这类工具选 Loop 是对的——它们服务的场景本身就没有"标准形状"，用户明天想要什么谁都不知道。

选 Graph——当任务形状已经清楚，你要的是可靠地跑一万次，而不是灵光乍现地跑一次。客服工单分诊、合同审查流水线、财报解读、发票核销、多方审批、有回滚需求的操作型 Agent。这时候你要的是可预测与可审计，牺牲一点"随机应变"是划算的。

更实际的做法是组合：外层是 Graph 定骨架，某个节点内部塞一个小 Loop 允许模型自由发挥——这样既有边界又有余地。2026 年下半年主流架构会长这个样子。

不是 Loop 输了，是 Loop 该退回它本来擅长的地方——探索；而生产线上，Graph 是把"能跑"变成"能维护"的关键动作。 — 结语
